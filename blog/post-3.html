Vibe Coding 2026.05.16

这个网站诞生记：我的 Vibe Coding 方法论 1.0

从一个空白文件夹到一个拥有双端适配、完整内容体系、自定义域名的个人网站，全程与 AI 对话完成。这篇是我对完整流程的复盘和方法论沉淀。

一、什么是 Vibe Coding？

Vibe Coding 不是"让 AI 帮你写代码"。Vibe Coding 是：你描述你想要的结果，AI 生成代码，你审查和迭代，直到满意。你和 AI 之间是一场持续的对话——你说"不对，我想要的是这样"，AI 修正，你再说"再调整一下那个"，AI 再修正。

听起来像魔法。但做了这个网站之后我发现，Vibe Coding 真正的门槛不在技术，而在于你能不能清楚描述你想要什么，以及你能不能判断 AI 给出的结果是否靠谱。

这两个能力，恰好是产品经理每天都在练的东西。

二、这个网站的全流程拆解

阶段 1：立项——明确你要做什么、给谁看、核心目标是什么

就像做产品需要 PRD，做网站也需要一个清晰的 brief。我花了一小时和 Claude Code 对话，明确了三件事：

  • 定位：个人简历网站，面向招聘方（HR 和业务负责人），展示产品实习经历、项目成果、产品思考
  • 调性：专业但不沉重，有设计感但不浮夸。一个产品经理的个人网站应该体现产品经理的审美
  • 核心目标：让招聘方在 3 分钟内看完我的经历，并觉得"这个人值得聊一聊"

这个阶段最重要的决策是：不追求炫技，追求信息传达效率。很多程序员做个人网站会放一堆动画和交互，但作为产品经理，我更需要的是内容被清晰、快速地被阅读。

阶段 2：框架——从信息架构到视觉骨架

确定了定位之后，我先和 AI 讨论了信息架构：几个板块、每个板块放什么、板块之间的顺序怎么排。

最终确定了 7 个板块：Hero → 教育背景 → 工作技能 → 实习经历 → 产品思考 → 科研项目 → 校园经历。这个顺序遵循了"最重要信息优先"的原则——招聘方最先想看到的是我是谁、我学过什么、我能做什么、我做过什么。

框架确定后，我让 AI 帮我搭建了 HTML 骨架，每个板块先用占位文字填充。这个阶段不用纠结细节，就像画原型图的低保真阶段——先确认结构是对的，再往里面填内容。

阶段 3：技术方案——为什么选这些技术？

技术栈是我和 AI 一起讨论确定的：

  • Tailwind CSS CDN：不需要构建工具，一个 HTML 文件搞定所有样式。原子化 CSS 让我可以快速调整布局，也方便 AI 理解和修改。
  • 原生 JS：没有用 React/Vue，因为这是一个单页展示型网站，不需要组件化和状态管理。原生 JS 更轻量，加载更快。
  • Lucide Icons：轻量、开源、风格统一的图标库，CDN 引入即可。
  • 自定义 CSS 变量：玻璃态效果、暗色模式、品牌色系统，全部通过 CSS 自定义属性管理。这在后面做暗色模式适配时省了巨大的工作量。

技术选型的第一原则：选择刚好够用的技术，而不是最酷的技术。一个 70KB 的页面不需要 React。

阶段 4：内容——AI 帮我组织文字，但故事是我的

内容是我自己写的。但 AI 帮了我很多忙：

  • 结构优化：我把一段流水账式的经历描述扔给 AI，它帮我拆成"背景 - 动作 - 成果"的子卡片结构
  • 数据提取：从我的口述中提取可量化的指标，形成标签展示
  • 文案精简：把啰嗦的长句改成适合网页阅读的短句

这个阶段我的体会是：AI 是你的编辑，但不是你的作者。故事、经历、洞见必须来自你自己。AI 可以帮你把故事讲得更清晰，但讲什么故事是你决定的。

阶段 5：结构调整——从 PC 端到移动端

网站第一版只做了 PC 端。部署之后我用手机打开看了一眼，阅读体验很差——布局太宽需要左右滑动，卡片间距过大，字体大小不合理。

于是我开始了第二阶段：移动端适配。

这个过程和做产品的"多端适配"逻辑完全一样：不是简单地把 PC 端缩小，而是为移动端重新设计信息层级和交互模式。

具体做了哪些？

  • 取消全屏高度：PC 端每个板块占满一屏，营造"翻页"的感觉。移动端改为内容驱动高度，让用户自然下滑浏览。
  • 轮播改手风琴：PC 端的实习经历是横向轮播，移动端改为纵向手风琴，点击展开。因为在手机上横向滑动体验很差，而且用户需要一览所有经历而不是一张一张翻。
  • 技能卡片压缩：移动端技能卡片高度减半，详情默认展开展示，取消了点击交互——因为用户不知道需要点击。
  • 间距和字号统一调优：全局收紧间距，调整字号层级。

移动端适配不是"让它在手机上也能看"，而是"让它在手机上更好看"。这是两种完全不同的标准。

阶段 6：细节优化——魔鬼在细节里

框架和内容就位后，我开始和 AI 逐页打磨细节：

  • 暗色模式：用 CSS 变量统一管理颜色，一键切换。所有玻璃态效果在暗色模式下重新调整了透明度和阴影。
  • 动画节奏：滚动入场动画、悬浮效果、过渡动画——每个动画都调整了曲线和时长，让页面"有感觉但不打扰"。
  • 文案微调：这是一轮又一轮的"改一个字、再看一眼"。括号里的内容、标签的措辞、时间的格式——这些琐碎的修改占了整个开发时间的 30%。

这个阶段最考验的是耐心和判断力。AI 可以快速生成新的版本，但"哪个版本更好"需要你来判断。有时候你试了三个方案都不满意，第四个才对了。这个过程和做产品设计迭代的逻辑一模一样。

阶段 7：部署——让网站真正上线

部署环节是整个过程中 AI 参与度最低的环节——因为涉及服务器、域名、SSL 证书等实际操作。但 AI 帮我把流程梳理得很清楚：

  1. 购买域名 → 2. 配置 DNS 解析 → 3. 部署到 GitHub Pages → 4. 配置 SSL 证书 → 5. 验证全链路上线

整个部署过程遇到的问题，我逐一和 AI 讨论解决方案。AI 就像一个随叫随到的技术顾问——它不能帮你按回车，但能告诉你每一步应该做什么、遇到报错怎么排查。

三、Vibe Coding 带给我的反思

1. 产品经理 + AI = 一个人的全栈团队

这个网站在传统开发模式下，至少需要一个前端开发 3-5 天的工作量。但通过和 AI 对话，我一个周末就完成了从 0 到上线的全过程。

作为产品经理，我一直处于"提需求但写不了代码"的尴尬境地。Vibe Coding 让我能够直接把自己的产品想法落地为可用的产品，而不需要等待开发资源。这个能力对产品经理来说是革命性的。

2. 但 AI 不是银弹

Vibe Coding 也有明显的局限：

  • AI 会犯错：会写出有 bug 的代码、会忘记你的上下文、会把之前修好的问题重新引入。你需要有基本的代码阅读能力来发现这些问题。
  • AI 不擅长整体架构：如果你不给它框架，它会写出一团能跑但难以维护的代码。需要你来确定架构方向。
  • 细节调整效率低："把这个颜色调亮一点""这个间距缩小 2px"——这些在代码里改一行的事，和 AI 对话可能要描述好几轮。

3. 版本管理和部署流程亟待优化

整个项目我通过 Git 管理版本，每次修改后 push 到 GitHub Pages 自动部署。但 Vibe Coding 的节奏太快了——你可能十分钟内改了 5 个地方，如果没有好的 commit 习惯，回滚会很痛苦。

后续我希望建立一个更规范的 Vibe Coding 工作流：每次对话有明确的 scope，一次只改一个模块，改了立刻 commit。

四、我的 Vibe Coding 方法论 1.0

总结这个过程，我提炼了一套初版方法论：

  1. 先有 Brief，再动手。明确你要做什么、给谁用、核心目标是什么。没有 brief 的 Vibe Coding 等于没有 PRD 的开发。
  2. 先搭骨架，再填肉。让 AI 先出整体框架（信息架构、页面结构），确认方向正确后再逐块细化。
  3. 一次只改一件事。不要在一个对话里同时让 AI 改布局、改内容、改颜色。分开改，改完立刻 commit。
  4. 你是产品经理，AI 是开发。你负责判断"什么是对的"，AI 负责实现。不要把判断权交给 AI。
  5. 移动端不是缩小版 PC 端。多端适配需要重新设计信息层级和交互模式，不能靠缩放解决。
  6. 部署和内容管理要提前规划。域名、SSL、图片资源——这些不是代码的事，但缺了它们网站就不完整。

这个网站会持续迭代。文章也会更新。但这套方法论 1.0，是我作为产品经理第一次用 Vibe Coding 完整交付一个产品的实践总结。

如果你也是产品经理，也想试试 Vibe Coding——开始吧。不需要会写代码，但你得知道你想做成什么样。
